Dashboard de Administración

Documentación técnica — Proyecto Joyería TFG
Fecha: 1 de Mayo de 2026

1. Descripción general

Se ha creado una miniweb de administración independiente bajo el prefijo /admin. El objetivo es separar las tareas internas de gestión de la tienda pública, evitando que los administradores tengan que entrar en páginas visibles para clientes para editar productos, pedidos o usuarios.

El dashboard inicial actúa como punto de entrada para el personal administrador. Incluye métricas básicas, acceso rápido a secciones internas y una navegación lateral propia. No utiliza el Header público de la tienda, sino un layout específico de administración.

2. Motivo del cambio

Antes existían pantallas de gestión mezcladas con el layout público, por ejemplo el CRUD de usuarios en /usuarios. Esto provocaba varios problemas:

  • La edición de datos internos se mostraba con el diseño de la tienda.
  • La navegación del administrador dependía de enlaces públicos.
  • No había una separación clara entre experiencia de cliente y panel interno.
  • Era más difícil ampliar funcionalidades como stock, pedidos, imágenes o métricas.

Con la nueva estructura, todo lo administrativo queda agrupado en /admin y protegido por autenticación y rol.

3. Control de acceso

Se ha creado un middleware para comprobar que el usuario autenticado tiene rol de administrador.

Archivo Función
app/Http/Middleware/AdminMiddleware.php Comprueba que el usuario está logeado y que Auth::user()->rol === 'admin'.
app/Http/Kernel.php Registra el alias admin para poder usarlo en rutas.
routes/web.php Agrupa las rutas internas con middleware(['auth', 'admin']), prefijo admin y nombres admin.*. 
Route::middleware(['auth', 'admin'])->prefix('admin')->name('admin.')->group(function () {
    Route::get('/dashboard', [DashboardController::class, 'index'])->name('dashboard');
});

4. Redirección tras login

Fortify redirigía a todos los usuarios a /, incluso si eran administradores. Para corregirlo, se creó una respuesta personalizada de login.

Archivo Función
app/Http/Responses/LoginResponse.php Si el usuario tiene rol admin, redirige a admin.dashboard. Si no, redirige a la tienda.
app/Providers/FortifyServiceProvider.php Registra la respuesta personalizada para el login de Fortify.
app/Http/Middleware/RedirectIfAuthenticated.php Si un administrador ya logeado intenta acceder a login o register, se le manda al dashboard.

5. Estructura de rutas

Ruta Nombre Descripción
/admin/dashboard admin.dashboard Vista inicial con métricas y accesos rápidos.
/admin/usuarios admin.usuarios.index Listado y búsqueda básica de usuarios.
/admin/productos admin.productos.index Listado, filtros básicos y stock rápido de productos.
/admin/pedidos admin.pedidos.index Listado y filtrado básico de pedidos.

6. Archivos principales creados

Archivo Función
app/Http/Controllers/Admin/DashboardController.php Controlador del dashboard. Calcula métricas como usuarios, productos, pedidos, stock bajo y ventas.
app/Http/Controllers/Admin/UserController.php CRUD administrativo de usuarios.
app/Http/Controllers/Admin/ProductoController.php CRUD administrativo de productos, subida de imagen y actualización rápida de stock.
app/Http/Controllers/Admin/PedidoController.php Gestión de pedidos, detalle y cambio de estado.
resources/views/layouts/admin.blade.php Layout aislado del panel con sidebar, estilos propios y sin header público.
resources/views/admin/ Carpeta con las vistas del dashboard, usuarios, productos y pedidos.

7. CRUD de usuarios

Se ha migrado la gestión de usuarios al panel interno. El CRUD antiguo situado en /usuarios se ha eliminado porque ya no se usa.

Funcionalidad actual

  • Listado de usuarios.
  • Búsqueda por nombre, apellidos o email.
  • Filtro por rol (admin o user).
  • Crear usuario.
  • Editar nombre, apellidos, email, teléfono, rol y contraseña.
  • Eliminar usuario.

Archivos eliminados

  • app/Http/Controllers/UserController.php
  • resources/views/usuarios/index.blade.php
  • resources/views/usuarios/create.blade.php
  • resources/views/usuarios/edit.blade.php
  • Ruta pública Route::resource('usuarios', UserController::class)

8. CRUD de productos

Se ha creado una gestión interna de productos separada de las páginas públicas de joyas.

Funcionalidad actual

  • Listado paginado de productos.
  • Búsqueda por nombre o marca.
  • Filtro por categoría.
  • Crear producto con imagen obligatoria.
  • Editar datos e imagen del producto.
  • Actualizar stock directamente desde el listado.
  • Eliminar producto y su imagen principal.

La imagen se guarda en el disco público de Laravel usando Storage::disk('public'), dentro de la carpeta productos.

9. Gestión de pedidos

Se ha añadido una zona de pedidos para administración, diferente del historial de pedidos del usuario cliente.

Funcionalidad actual

  • Listado paginado de pedidos.
  • Filtro básico por estado.
  • Detalle del pedido con productos, cantidades, precio unitario y subtotal.
  • Cambio de estado del pedido: pendiente, preparado, enviado, entregado o cancelado.
  • Eliminación de pedido.

10. Diseño del dashboard

El dashboard usa un layout propio con sidebar lateral y contenido central. Se decidió no usar el header de la tienda porque el panel de administración necesita una navegación más operativa, enfocada a gestión repetida y no a compra.

/admin/dashboard
├── Usuarios
├── Productos
├── Pedidos
└── Ver tienda

Los estilos base del admin se han incluido temporalmente en resources/views/layouts/admin.blade.php. Esto se hizo porque durante el desarrollo se detectó que la recompilación de Sass con Laravel Mix no siempre actualizaba public/css/app.css por problemas con node_modules entre Windows, WSL y Docker.

11. Funcionalidades pendientes

El dashboard actual es una base funcional. Quedan pendientes varias mejoras para convertirlo en una herramienta completa de gestión:

  • Filtros avanzados de productos: por precio, stock, género, material, color, fecha de alta y productos sin imagen.
  • Buscador global: una búsqueda única en el sidebar o cabecera del admin para encontrar usuarios, pedidos o productos.
  • Ordenación de tablas: permitir ordenar por precio, stock, fecha, estado o cliente.
  • Gestión de varias imágenes: usar la tabla IMAGENES_PRODUCTO para permitir galería, imagen principal y eliminación individual.
  • Validaciones más finas de stock: avisos visuales por stock bajo, stock agotado y reposición recomendada.
  • Panel de pedidos más completo: dirección de envío, método de pago, historial de cambios de estado y notas internas.
  • Protección contra borrados críticos: impedir que un administrador se elimine a sí mismo o que se borre el último admin.
  • Dashboard de métricas reales: ventas por periodo, productos más vendidos, pedidos pendientes y evolución mensual.
  • Exportación: descargar productos, usuarios o pedidos en CSV/Excel.
  • Tests: pruebas de acceso admin, CRUD y permisos.

12. Consideraciones técnicas

  • Todo lo nuevo del panel se ha colocado bajo el namespace App\Http\Controllers\Admin.
  • Las vistas nuevas viven en resources/views/admin.
  • Las rutas administrativas usan nombres admin.* para no mezclarse con rutas públicas.
  • El CRUD antiguo de usuarios se eliminó para evitar duplicidad y confusión.
  • El acceso al panel depende de que el campo rol del usuario sea exactamente admin.
