Imágenes de Producto con AWS S3

Documentación técnica — Proyecto Joyería TFG
Fecha: 7 de Mayo de 2026

1. Descripción general

Se ha modificado el sistema de imágenes de producto para utilizar la tabla IMAGENES_PRODUCTO y almacenar los archivos en un bucket de AWS S3. Esta mejora permite que cada producto tenga una o varias imágenes, manteniendo una imagen principal automática.

La funcionalidad de grabados personalizados no se ha modificado. Los grabados continúan guardándose en storage/app/public/grabados mediante el campo ruta_grabado_personalizado de carrito y detalles de pedido.

  • Las imágenes de producto se suben a S3 dentro del directorio lógico productos/.
  • La primera imagen subida se marca automáticamente como principal = true.
  • Las vistas consumen la URL mediante el atributo imagen_principal_url.
  • La tabla IMAGENES_PRODUCTO guarda la ruta/clave del objeto S3 en el campo url.

3. Configuración de AWS S3

Laravel ya dispone de un disco s3 en config/filesystems.php. Para conectarlo con AWS S3 real hay que completar las variables del archivo .env:

AWS_ACCESS_KEY_ID=clave_access_key
AWS_SECRET_ACCESS_KEY=clave_secret_key
AWS_DEFAULT_REGION=eu-west-3
AWS_BUCKET=nombre-del-bucket
AWS_URL=https://nombre-del-bucket.s3.eu-west-3.amazonaws.com
AWS_USE_PATH_STYLE_ENDPOINT=false

En caso de utilizar un servicio compatible con S3, como MinIO, también se configuraría AWS_ENDPOINT y normalmente AWS_USE_PATH_STYLE_ENDPOINT=true. Para AWS S3 real no suele ser necesario indicar endpoint manualmente.

3.1 Dependencia necesaria

En Laravel 8 el disco S3 utiliza Flysystem v1. Por eso se añadió la dependencia:

composer require league/flysystem-aws-s3-v3:^1.0

Si el entorno ya tiene modificado composer.json, se puede actualizar el lock con:

composer update league/flysystem-aws-s3-v3 --with-dependencies

4. Modelo de datos

La tabla IMAGENES_PRODUCTO ya existía en el proyecto y contiene los campos necesarios:

Campo Uso
id Identificador único de la imagen.
url Ruta o clave del objeto dentro de S3, por ejemplo productos/abc123.webp.
principal Indica si la imagen es la principal del producto.
id_producto Clave foránea hacia la tabla PRODUCTO.

Se recomienda guardar en base de datos la ruta del objeto S3 y no la URL pública completa. Así, si en el futuro se cambia el dominio, bucket o se añade CloudFront, no será necesario migrar las URLs guardadas.

5. Relaciones y atributos del modelo

5.1 Relación de imágenes

public function imagenes()
{
    return $this->hasMany(ImagenProducto::class, 'id_producto');
}

5.2 Imagen principal

public function imagenPrincipal()
{
    return $this->hasOne(ImagenProducto::class, 'id_producto')
        ->where('principal', true);
}

5.3 URL final para las vistas

Las vistas no acceden directamente al disco S3, sino que llaman al atributo:

$producto->imagen_principal_url

Este atributo busca primero una imagen marcada como principal. Si no existe, utiliza la primera imagen relacionada. Como compatibilidad temporal, si el producto todavía no tiene filas en IMAGENES_PRODUCTO, puede devolver la antigua ruta ruta_grabado.

6. Flujo de subida de imágenes

Admin selecciona imágenes
↓
Formulario envía imagenes[]
↓
Laravel valida tipo y tamaño
↓
Se sube cada archivo a Storage::disk('s3')
↓
Se crea una fila en IMAGENES_PRODUCTO
↓
La primera imagen queda marcada como principal

El campo HTML se cambió para permitir múltiples archivos:

<input type="file" id="imagenes" name="imagenes[]" accept="image/*" multiple>

La validación del controlador permite uno o varios archivos:

'imagenes' => 'nullable|array|min:1',
'imagenes.*' => 'image|mimes:jpeg,jpg,png,gif,webp|max:2048',

Al crear un producto desde administración, las imágenes son obligatorias. Al editar, son opcionales; si se suben nuevas imágenes, sustituyen a las anteriores.

7. Lectura de imágenes en vistas

Antes se usaba directamente asset('storage/' . $producto->ruta_grabado). Ahora las vistas utilizan el atributo centralizado:

@if($producto->imagen_principal_url)
    <img src="{{ $producto->imagen_principal_url }}" alt="{{ $producto->nombre }}">
@else
    <i class="bi bi-gem"></i>
@endif

Esto evita repetir lógica de almacenamiento en cada Blade. Si el origen de las imágenes cambia en el futuro, solo será necesario modificar el modelo.

8. Separación con grabados personalizados

Los grabados personalizados no forman parte de la tabla IMAGENES_PRODUCTO. Se mantienen en campos independientes:

  • CARRITO.ruta_grabado_personalizado
  • DETALLES_PEDIDO.ruta_grabado_personalizado

El controlador PersonalizaController sigue guardando los grabados con:

Storage::disk('public')->put($nombreArchivo, $imagenBinaria);

Por tanto, las imágenes comerciales del producto se sirven desde S3, mientras que los diseños personalizados creados por el usuario permanecen en el almacenamiento público local de Laravel.

9. Consideraciones de permisos

Para una tienda online, lo más práctico es que las imágenes de producto sean públicas de lectura. Esto no significa publicar las claves de AWS, sino permitir que el navegador pueda cargar las imágenes subidas al bucket o al prefijo productos/.

Una política típica del bucket permitiría lectura pública solo de objetos:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::nombre-del-bucket/productos/*"
    }
  ]
}

La escritura debe quedar limitada al usuario IAM usado por Laravel, con permisos sobre el bucket o prefijo necesario. Las credenciales nunca deben subirse al repositorio.

10. Pruebas recomendadas

  • Crear un producto desde el panel admin con una sola imagen.
  • Crear un producto con varias imágenes y comprobar que la primera aparece como principal.
  • Editar un producto y subir nuevas imágenes para verificar que sustituyen a las anteriores.
  • Comprobar que catálogo, detalle, carrito, favoritos y pedidos muestran la imagen principal.
  • Crear un grabado personalizado y verificar que se sigue guardando y mostrando desde storage.
  • Revisar en AWS S3 que los archivos se han creado dentro de productos/.

11. Resumen final

Con esta implementación, el sistema de imágenes de producto queda preparado para trabajar con AWS S3 y con múltiples imágenes por producto. La tabla IMAGENES_PRODUCTO pasa a ser la fuente principal de las fotografías comerciales, mientras que ruta_grabado_personalizado mantiene intacto el flujo de grabados personalizados.

La mejora centraliza la lectura de imágenes en el modelo Producto, reduce dependencias de rutas locales en las vistas y facilita una futura integración con dominios propios o CloudFront.
